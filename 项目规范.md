# 项目规范文档

## 📋 概述

本文档定义了软考文档汇总项目的各项规范，包括`Git提交规范`、`文档格式规范`、`命名规范`、`内容质量规范`等，确保项目的统一性和可维护性。

## 1 🔄 Git 提交规范

### 📝 Git commit message 格式

采用 [Conventional Commits](https://www.conventionalcommits.org/zh-hans/v1.0.0/) 规范：

```
<type>[optional scope]: <description>

[optional body]
```
### 🏷️ 提交类型 (type)

| 类型           | 说明           | 示例                     |
|---------------|---------------|--------------------------|
| **feat**     | 新功能          | `feat: 添加2024年真题解析`    |
| **fix**      | 修复bug        | `fix: 修复链接失效问题`        |
| **docs**     | 文档更新         | `docs: 更新README.md说明`  |
| **style**    | 格式调整         | `style: 统一markdown格式`  |
| **refactor** | 重构           | `refactor: 优化目录结构`     |
| **perf**     | 性能优化         | `perf: 优化图片加载速度`       |
| **test**     | 测试相关         | `test: 添加链接检查脚本`       |
| **chore**    | 构建过程或辅助工具的变动 | `chore: 更新依赖包`         |
| **revert**   | 回滚           | `revert: 回滚feat: 新增功能` |

### 🎯 提交范围 (scope)

- **video**: 视频相关内容
- **material**: 教材资料
- **exam**: 真题练习
- **paper**: 论文写作
- **notes**: 讲义笔记
- **knowledge**: 知识点梳理
- **common**: 软考常识
- **mental**: 思想准备
- **prep**: 考试准备
- **tools**: 工具软件
- **docs**: 项目文档
- **config**: 配置文件

### 📖 提交示例

```bash
# 新功能
git commit -m "feat(exam): 添加2024年上半年真题解析"

# 修复问题
git commit -m "fix: 修复README.md中的失效链接"

# 文档更新
git commit -m "docs(knowledge): 更新计算机基础知识章节"

# 格式调整
git commit -m "style: 统一所有markdown文件的标题格式"

# 重构
git commit -m "refactor: 优化真题目录结构，按年份分类"
```

### 📋 提交信息规范

- **使用中文**：提交信息使用中文描述，便于团队理解
- **简洁明确**：描述要简洁明了，突出重点
- **动词开头**：使用动词开头，如"添加"、"修复"、"更新"
- **避免重复**：避免在描述中重复提交类型的信息
- **详细说明**：复杂修改需要在body中详细说明


## 2 📄 Markdown 文档规范

### 2.1 🏗️ 文档结构

每个Markdown文档应包含以下结构：

```md
---
title: 文档标题
subject: 所属科目
category: 文档分类
tags: [标签1, 标签2]
difficulty: 初级/中级/高级
created: 2024-01-01
updated: 2024-01-01
---

# 文档标题

## 概述
[文档简要介绍]

## 主要内容
[详细内容]

## 相关链接
- [[相关文档1]]
- [[相关文档2]]

## 参考资料
[参考资料列表]
```

### 2.2 📝 标题规范  

- **层级清晰**：使用 # ## ### #### ##### ###### 表示1-6级标题
- **标题格式**：标题后不加空格，直接写内容
- **避免跳级**：不要跳过标题层级，如从 # 直接跳到 ###
- **唯一性**：同一文档中同级标题不能重复

### 2.3 📊 表格规范

```markdown
| 列1 | 列2 | 列3 |
| --- | --- | --- |
| 内容1 | 内容2 | 内容3 |
| 内容4 | 内容5 | 内容6 |
```

- **对齐方式**：左对齐（默认）、右对齐（:-:）、居中（:-:）
- **表头明确**：表头要清晰表达列的含义
- **内容简洁**：表格内容要简洁明了

### 2.4 🔗 链接规范

#### 2.4.1 内部链接
```markdown

[[文档名称]]

# 带别名的链接
[[文档名称|别名]]

# 跨文件夹链接
[[文件夹/文档名称]]

# 嵌入文档
![[文档名称]]
```

#### 2.4.2 外部链接
```markdown
# 普通链接
[链接文本](URL)

# 带标题的链接
[链接文本](URL "链接标题")
```

#### 2.4.3引用链接
```markdown
[链接文本][引用标识]

```

## 3 📁 文件命名规范

### 3.1 📂 目录命名

#### 3.1.1 主目录格式
```
序号. 分类【描述】/
```

**示例**：
- `1. 视频【最新全套视频】/`
- `2. 教材【最新教材+大纲+指导】/`
- `9.1. 其他备考资料/`

#### 3.1.2 子目录格式
```
序号_内容名称/
```

**示例**：
- `01_基础知识视频/`
- `02_官方教材/`
- `03_历年真题/`

### 3.2 📄 文件命名

#### 3.2.1 Markdown文件
```
描述性名称.md
```

**示例**：
- `exam_outline_2024.md`
- `knowledge_points_summary.md`
- `study_plan_template.md`

#### 3.2.2 图片文件
```
科目_类别_描述_序号.扩展名
或
科目_类别_描述_日期时间(yyyyMMdd_HHmmss).扩展名
```

**组成部分说明**：
- **科目**：科目简称（arch、network、pm、analyst、common）
- **类别**：图片类型（diagram、chart、screenshot、mindmap、photo、icon）
- **描述**：图片内容的简洁英文描述，使用下划线分隔
- **序号**：两位数字，从01开始（用于系列图片）
- **日期时间**：精确到分钟的时间戳（用于唯一标识）
- **扩展名**：png、jpg、svg、gif

**命名方式选择**：
- **序号方式**：适用于系列图片、多步骤图表
- **日期时间方式**：适用于截图、临时图片、唯一性要求高的图片

### 3.3 🖼️ 图片管理规范

#### 3.3.1 📁 存储结构
```
images/
├── arch_diagram_layer_architecture_01.png
├── arch_chart_performance_comparison_01.svg
├── network_diagram_network_topology_01.png
├── pm_screenshot_project_dashboard_01.png

├── common_photo_study_materials_01.jpg
└── ...
```

#### 3.3.2 📏 技术要求

**文件格式选择**：
- **PNG**：适用于架构图、流程图、截图（支持透明背景）
- **SVG**：适用于图表、图标（矢量格式，可缩放）
- **JPG**：适用于照片、实物图（文件较小）
- **GIF**：适用于简单动画、演示

**尺寸规范**：

| 类型 | 最大宽度 | 建议尺寸 | 说明 |
|------|----------|----------|------|
| 架构图 | 1200px | 800x600以上 | 清晰展示结构 |
| 图表 | 1000px | 600x400以上 | 数据展示清晰 |
| 截图 | 1200px | 原始尺寸 | 保持原始比例 |
| 图标 | 128px | 64x64或128x128 | 小尺寸图标 |
| 照片 | 1200px | 800x600以上 | 高质量照片 |

**文件大小限制**：

| 文件类型 | 最大大小 | 说明 |
|----------|----------|------|
| 单个图片 | 2MB | PNG、JPG、GIF格式 |
| SVG文件 | 500KB | 矢量图形文件 |
| 图标文件 | 100KB | 小尺寸图标 |
| 照片文件 | 1MB | 高质量照片 |

#### 3.3.3 🎨 质量标准

**清晰度要求**：

| 项目 | 要求 | 说明 |
|------|------|------|
| 分辨率 | 至少72dpi，推荐150dpi | 确保图片清晰度 |
| 色彩模式 | RGB模式 | 适用于屏幕显示 |
| 对比度 | 足够 | 确保文字和背景对比度 |
| 可读性 | 清晰可读 | 文字清晰，线条明显 |

**内容规范**：

| 项目 | 要求 | 说明 |
|------|------|------|
| 主题明确 | 与文档主题相关 | 图片内容要相关 |
| 信息完整 | 包含必要信息 | 包含必要的信息和标注 |
| 风格统一 | 同类型风格一致 | 保持风格一致性 |
| 版权合规 | 有版权许可或原创 | 避免版权问题 |

### 4.1 📋 维护规范

**添加新图片**：
1. **命名检查**：确保符合命名规范
2. **格式检查**：选择合适的文件格式

4. **大小检查**：确保文件大小符合要求

**定期维护**：
- **每月检查**：检查失效的图片引用
- **季度优化**：压缩过大的图片文件
- **半年清理**：删除未使用的图片文件
- **年度整理**：重新组织和优化图片结构

**版本控制**：
- **Git LFS**：大文件使用Git LFS管理
- **忽略文件**：在.gitignore中忽略临时图片文件
- **提交规范**：图片提交使用规范的commit信息

### 4.2 🛠️ 工具推荐

**图片编辑工具**：
- **免费工具**：GIMP、Inkscape、Paint.NET
- **在线工具**：Canva、Figma、Photopea
- **专业工具**：Adobe Photoshop、Illustrator

**图片优化工具**：
```bash
# PNG压缩
pngquant --quality=65-80 images/*.png

# JPEG压缩
jpegoptim --max=80 images/*.jpg

# SVG优化
svgo images/*.svg
```

**批量处理脚本**：
```bash
#!/bin/bash
# 批量重命名图片文件
for file in *.png; do
    # 根据实际需求重命名
    mv "$file" "new_name.png"
done
```

### 4.3 ⚠️ 注意事项

**版权问题**：

- **懒加载**：网页中使用懒加载技术

- **缓存策略**：设置合适的缓存策略
- **响应式**：支持不同设备的显示需求

**备份策略**：
- **云端备份**：重要图片云端备份
- **本地备份**：本地保留原始文件
- **版本备份**：保留重要版本的历史备份

### 4.11 📊 版本管理规范

### 🏷️ 版本号规范

采用 [语义化版本](https://semver.org/lang/zh-CN/) 规范：

```
主版本号.次版本号.修订号
```

- **主版本号**：不兼容的API修改
- **次版本号**：向下兼容的功能性新增

   - 编写发布说明

   - 通知用户
